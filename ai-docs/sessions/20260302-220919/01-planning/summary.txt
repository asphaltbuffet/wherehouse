The plan adds a `//go:generate stringer -type=EventType -linecomment` directive to `internal/database/eventTypes.go` so each constant (e.g. `ItemCreatedEvent`) produces its dot-separated string (e.g. `"item.created"`), then adds hand-written `Value()`/`Scan()` and `ParseEventType()` in a new `eventTypes_sql.go` to bridge the typed int to the database string column. The `Event.EventType` field, `AppendEvent`/`insertEvent`/`GetEventsByType` signatures, and the `eventHandler.go` switch are all updated to use the typed enum, with call sites in `cmd/` and `internal/cli/` migrated to named constants in strict TDD order (unit tests for `ParseEventType`, `String()`, and `Value`/`Scan` written first, failing, then implementations added). The `internal/styles.EventStyle` method remains `string`-based and receives `.String()` at each call site to avoid coupling the styles package to the database package.
